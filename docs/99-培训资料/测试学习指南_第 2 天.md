## 二、测试代码的命名规则

### 2.1 文件命名规则

**规则：** `test_功能模块.py`

**例子：**

```
✅ test_auth.py          # 认证模块测试
✅ test_breed.py         # 品种管理测试
✅ test_charts.py        # 图表功能测试
❌ AuthTest.py           # 错误：不符合命名规范
❌ test.py               # 错误：没有体现功能模块
```

**为什么这样命名？**

- pytest 会自动识别 `test_` 开头的文件
- 看到文件名就知道是测什么的
- 方便管理和查找

**小技巧：**

```bash
# 列出所有测试文件
dir /b Test\test_*.py
```

### 2.2 类的命名规则

**规则：** `Test + 功能模块名 + Functionality/Tests`

**例子：**

```python
# ✅ 正确的命名
class TestUserAuthentication:      # 用户认证测试类
class TestBreedManagement:         # 品种管理测试类
class TestChartsFunctionality:     # 图表功能测试类

# ❌ 错误的命名
class AuthTest:                    # 不符合规范
class Test:                        # 太简单，不知道测什么
```

**记忆技巧：**

- 必须以 `Test` 开头（大写 T）
- 中间是功能模块名（驼峰命名）
- 可以加 `Functionality` 或 `Tests` 后缀

### 2.3 方法的命名规则

**规则：** `test_ + 具体测试场景`

**例子：**

```python
# ✅ 清晰的命名
def test_login_success(self):              # 登录成功
def test_register_duplicate_username(self) # 注册重复用户名
def test_sql_injection_attack(self)        # SQL 注入攻击

# ❌ 糟糕的命名
def test1(self):                           # 不知道测什么
def login(self):                           # 缺少 test_前缀
def testLogin(self):                       # 应该用下划线分隔
```

**最佳实践：**

- 用 `test_` 开头
- 全部小写
- 单词间用下划线 `_` 分隔
- 名字要能清楚表达测试场景

## 三、fixture 详解（重点！）

### 3.1 什么是 fixture？

**通俗解释：**
fixture = **测试前的准备工作**

就像做饭前要买菜、洗菜一样，测试前也要做准备：

- 准备测试数据
- 创建数据库连接
- 模拟浏览器
- 创建测试用户

**官方定义：**

> Fixture 是 pytest 提供的一种依赖注入机制，用于在测试执行前设置测试环境。

### 3.2 常见的 fixture

**本项目中的 fixture：**

| Fixture 名称         | 作用         | 来源文件        |
| ------------------ | ---------- | ----------- |
| `client`           | 模拟浏览器      | conftest.py |
| `db`               | 数据库会话      | conftest.py |
| `logged_in_client` | 已登录的浏览器    | conftest.py |
| `app`              | Flask 应用实例 | conftest.py |

### 3.3 client fixture 详解

**作用：** 模拟浏览器访问网站

**不用 fixture 的写法（麻烦）：**

```python
from app import app

def test_login():
    # 每次都要手动创建客户端
    client = app.test_client()

    # 手动启动应用
    with app.app_context():
        response = client.post('/login', data={
            'username': 'test',
            'password': '123456'
        })

        assert response.status_code == 200
```

**使用 fixture 的写法（简单）：**

```python
def test_login(client):  # ← 直接用！
    """client 会自动创建"""
    response = client.post('/login', data={
        'username': 'test',
        'password': '123456'
    })

    assert response.status_code == 200
```

**对比：**

- 不用 fixture：10 行代码
- 使用 fixture：5 行代码
- **节省了 50% 的代码量！**

### 3.4 db fixture 详解

**作用：** 提供数据库会话，自动清理数据

**实际例子：**

```python
def test_user_registration(client, db):
    """
    client → 模拟浏览器
    db     → 数据库会话（自动创建和清理）
    """
    # 使用 db 创建用户
    user = User(username='testuser')
    user.set_password('password123')
    db.session.add(user)
    db.session.commit()

    # 测试结束后，db fixture 会自动清理数据
    # 不需要你手动删除！
```

**自动清理的好处：**

- ✅ 测试之间互不影响
- ✅ 不会污染数据库
- ✅ 每次测试都是干净的环境

### 3.5 fixture 在哪定义的？

**查看 conftest.py 文件：**

打开文件：`conftest.py`（在项目根目录）

你会看到类似这样的代码：

```python
import pytest
from app import create_app
from models import db as database

@pytest.fixture
def app():
    """创建 Flask 应用实例"""
    app = create_app()

    with app.app_context():
        database.create_all()  # 创建所有表
        yield app              # 返回给测试使用
        database.drop_all()    # 测试完后删除所有表


@pytest.fixture
def client(app):
    """创建测试客户端"""
    return app.test_client()


@pytest.fixture
def db(app):
    """创建数据库会话"""
    with app.app_context():
        yield database.session
```

**代码解析：**

```python
# 第 1 步：导入需要的模块
import pytest
from app import create_app
from models import db as database

# 第 2 步：定义 fixture
@pytest.fixture  # ← 这个装饰器告诉 pytest 这是一个 fixture
def app():
    """创建 Flask 应用实例"""
    # 准备阶段
    app = create_app()  # 创建应用

    with app.app_context():
        database.create_all()  # 创建数据库表

        # 返回给测试使用
        yield app  # ← yield 后面的代码测试完后执行

        # 清理阶段
        database.drop_all()  # 删除数据库表


@pytest.fixture
def client(app):
    """创建测试客户端"""
    # app 参数会自动调用上面的 app fixture
    return app.test_client()


@pytest.fixture
def db(app):
    """创建数据库会话"""
    with app.app_context():
        yield database.session  # 返回数据库会话对象
```

**yield 关键字的作用：**

```python
yield app  # 分成两部分理解：

# yield 之前：准备阶段（测试前执行）
- 创建应用
- 创建数据库表

# yield 之后：清理阶段（测试后执行）
- 删除数据库表
- 释放资源
```

### 3.6 如何使用 fixture？

**方法 1：作为函数参数（最常用）**

```python
def test_something(client, db):
    """
    参数名要和 fixture 名一致
    client → 使用 client fixture
    db     → 使用 db fixture
    """
    # 直接使用
    response = client.get('/some-page')
```

**方法 2：使用装饰器**

```python
@pytest.mark.usefixtures("client", "db")
def test_something():
    """
    用装饰器指定使用哪些 fixture
    """
    # 但这样不能直接使用，要用 request 参数
    pass
```

**推荐使用方法 1**，更简单直观！

## 四、装饰器 @test_case 详解

### 4.1 什么是装饰器？

**通俗解释：**
装饰器 = **包装器**

就像给礼物包装一样，装饰器给函数"包装"一些额外功能。

**生活中的例子：**

```
原始功能：开车
装饰器：导航系统
组合后：开车 + 导航 = 会导航的开车

原始功能：拍照
装饰器：美颜滤镜
组合后：拍照 + 美颜 = 会美颜的拍照
```

**代码中的例子：**

```python
# 原始函数
def login():
    print("登录功能")

# 加上日志装饰器
@log_decorator
def login():
    print("登录功能")
    # 现在会自动记录日志了！
```

### 4.2 @test_case 装饰器的作用

**作用：** 给测试函数添加元数据

**元数据包括：**

- 测试用例编号（如 TC-AUTH-001）
- 优先级（Critical/High/Medium/Low）
- 预期结果描述
- 测试类型

### 4.3 查看 @test_case 的实现

打开文件：`Test/test_framework.py`

找到 `test_case` 函数的定义：

```python
def test_case(case_id, priority='Medium', expected='', test_type='功能测试'):
    """
    测试用例装饰器

    参数:
        case_id: 测试用例编号，如 'TC-AUTH-001'
        priority: 优先级，默认 Medium
        expected: 预期结果描述
        test_type: 测试类型，默认功能测试
    """
    def decorator(func):
        # 把测试信息绑定到函数上
        func._test_case_id = case_id
        func._test_priority = priority
        func._test_expected = expected
        func._test_type = test_type

        # 返回原函数
        return func

    return decorator
```

**代码解析：**

```python
# 第 1 层：接收装饰器参数
def test_case(case_id, priority='Medium', expected='', test_type='功能测试'):
    # case_id='TC-AUTH-001'
    # priority='High'
    # expected='注册成功'
    # test_type='功能测试'

    # 第 2 层：真正的装饰器
    def decorator(func):
        # func 是被装饰的函数（如 test_register_success）

        # 给函数添加属性
        func._test_case_id = case_id       # 添加 ID
        func._test_priority = priority     # 添加优先级
        func._test_expected = expected     # 添加预期结果
        func._test_type = test_type        # 添加测试类型

        return func  # 返回原函数

    return decorator
```

**使用示例：**

```python
# 不使用装饰器
def test_register_success():
    pass

# 手动添加属性（麻烦）
test_register_success._test_case_id = 'TC-AUTH-001'
test_register_success._test_priority = 'High'

# 使用装饰器（简单）
@test_case('TC-AUTH-001', priority='High', expected='注册成功')
def test_register_success():
    pass
```

### 4.4 装饰器执行流程

**执行顺序：**

```python
# 代码执行顺序
1. Python 解释器遇到 @test_case(...)
2. 执行 test_case(...) 函数，返回 decorator
3. 用 decorator 包装 test_register_success 函数
4. 给函数添加属性：_test_case_id, _test_priority 等
5. 返回包装后的函数

# 测试执行时
pytest 收集测试 → 发现函数有 _test_case_id 属性 
→ 知道这是测试用例 → 执行测试 → 记录结果
```

## 五、完整的测试代码解析

### 5.1 逐行分析

让我们详细分析一个完整的测试：

```python
# ========== 导入部分 ==========
from .test_framework import test_case, test_manager, TestResult
# 从 test_framework 模块导入三个工具
# test_case    → 装饰器，标记测试用例
# test_manager → 管理器，记录测试结果
# TestResult   → 结果类，创建结果对象


# ========== 测试类定义 ==========
class TestUserAuthentication:
    """用户认证测试类"""
    # 类名：Test + UserAuthentication
    # 文档字符串：说明这个类测什么

    # ========== 测试方法定义 ==========
    @test_case('TC-AUTH-001', priority='High', expected='注册成功，跳转到登录页面')
    def test_register_success(self, client, db):
        """用户注册 - 成功场景"""

        # ========== 创建结果对象 ==========
        result = TestResult('TC-AUTH-001', 'test_register_success', '用户认证', 'High')
        result.expected_result = '注册成功，创建新用户'
        # TestResult(用例 ID, 方法名，模块名，优先级)
        # 创建结果对象，用来记录测试结果

        # ========== try-finally 结构 ==========
        try:
            # ====== 执行测试 ======

            # 1. 模拟浏览器发送 POST 请求
            response = client.post('/register', data={
                'username': 'testuser001',
                'password': 'TestPass123!'
            }, follow_redirects=True)
            # client.post() → 模拟提交表单
            # data={}       → 表单数据
            # follow_redirects=True → 自动跟随重定向

            # 2. 获取响应文本
            response_text = response.get_data(as_text=True)

            # 3. 断言验证
            assert response.status_code == 200
            # 状态码必须是 200（成功）

            assert '登录' in response_text
            # 页面必须包含"登录"二字（说明跳转到了登录页）

            # 4. 验证数据库
            user = User.query.filter_by(username='testuser001').first()
            assert user is not None
            # 数据库中必须有这个用户
            assert user.username == 'testuser001'
            # 用户名必须正确

            # ====== 记录成功 ======
            result.status = 'PASS'
            result.actual_result = f'注册成功，用户 ID: {user.id}'

        except AssertionError as e:
            # ====== 处理失败 ======
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'注册失败：{str(e)}'
            raise  # 重新抛出异常，让 pytest 知道测试失败了

        finally:
            # ====== 无论成败都要执行 ======
            test_manager.record_result(result)
            # 记录测试结果到报告
```

### 5.2 代码结构图

```
测试文件结构
│
├─ 导入语句
│   └─ from .test_framework import ...
│

"""
首页功能测试用例
"""
from .test_framework import test_case, test_manager, TestResult


class TestHomepageFunctionality:
    """首页功能测试类"""

    @test_case('TC-HOME-001', priority='Medium', expected='首页正常加载')
    def test_homepage_loads(self, client):
        """测试首页加载"""

        # 创建结果对象
        result = TestResult('TC-HOME-001', 'test_homepage_loads', '首页功能', 'Medium')
        result.expected_result = '首页正常加载，状态码 200'

        try:
            # 访问首页
            response = client.get('/')

            # 验证状态码
            assert response.status_code == 200

            # 验证页面内容
            response_text = response.get_data(as_text=True)
            assert '首页' in response_text or 'Home' in response_text or '数据看板' in response_text

            # 记录成功
            result.status = 'PASS'
            result.actual_result = '首页加载成功'

        except AssertionError as e:
            # 处理失败
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'首页加载失败：{str(e)}'
            raise

        finally:
            # 记录结果
            test_manager.record_result(result)
```

## 七、常见错误与解决方案

### 错误 1：找不到 fixture

**错误信息：**

```
fixture 'client' not found
```

**原因：**

- conftest.py 不在正确的位置
- conftest.py 里没有定义 client fixture

**解决方案：**

1. 检查 conftest.py 是否在 `Test/conftest.py` 或项目根目录
2. 确认里面有 `@pytest.fixture def client(): ...`

### 错误 2：装饰器参数错误

**错误信息：**

```
TypeError: test_case() missing required argument 'case_id'
```

**原因：**
忘记传 case_id 参数

**错误示例：**

```python
@test_case(priority='High')  # ❌ 缺少 case_id
def test_something():
    pass
```

**正确示例：**

```python
@test_case('TC-XXX-001', priority='High')  # ✅ 有 case_id
def test_something():
    pass
```

### 错误 3：断言总是失败

**错误信息：**

```
AssertionError: assert '登录' in '<html>...</html>'
```

**可能原因：**

1. 页面真的没有这个词
2. 编码问题导致中文乱码

**调试方法：**

```python
response_text = response.get_data(as_text=True)
print(response_text)  # 打印出来看看
assert '登录' in response_text
```

## 九、知识总结

### 今日重点：

1. ✅ 文件命名：`test_功能.py`
2. ✅ 类命名：`Test + 功能名`
3. ✅ 方法命名：`test_场景`

5. ✅ **装饰器 = 给函数添加元数据**
6. ✅ 完整的测试代码结构

### 核心概念：

- **Fixture**：pytest 的依赖注入，自动准备测试环境
- **@pytest.fixture**：定义 fixture 的装饰器
- **yield**：分隔准备阶段和清理阶段
- **@test_case**：自定义装饰器，添加测试元数据
- **try-finally**：确保测试结果被记录

### 难点解答：

**Q1: fixture 的执行顺序是怎样的？**

> A: pytest 会先执行所有 fixture 的 yield 前代码，然后执行测试，最后执行 fixture 的 yield 后代码。

**Q2: 多个 fixture 之间有依赖关系怎么办？**

> A: 在被依赖的 fixture 参数里写上依赖的 fixture 名。比如 `def client(app):` 表示 client 依赖 app。

**Q3: 为什么装饰器要返回 decorator 函数？**

> A: 因为装饰器需要接收参数（如 case_id、priority），所以要分两层：外层接收装饰器参数，内层接收被装饰的函数。

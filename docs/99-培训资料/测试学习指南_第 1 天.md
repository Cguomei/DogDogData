## 一、什么是测试用例？

### 1.1 通俗理解

**测试用例 = 考试题**

就像考试有题目一样，软件测试也有"考题"。每道考题包括：

```
题目编号：TC-AUTH-001
题目名称：用户注册 - 成功场景
优先级：High（重要）
操作步骤：
  1. 打开注册页面
  2. 输入用户名：testuser001
  3. 输入密码：TestPass123!
  4. 点击"注册"按钮

预期结果：
  ✓ 显示"注册成功"
  ✓ 跳转到登录页面
  ✓ 数据库中多了一个用户
```

### 1.2 测试用例的组成要素

看一个实际例子（来自项目）：

**文件位置：** `Test/docs/测试用例执行记录表_用户认证.txt`

打开文件，你会看到这样的表格：

| 字段       | 说明      | 示例                             |
| -------- | ------- | ------------------------------ |
| **ID**   | 测试用例编号  | TC-AUTH-001                    |
| **优先级**  | 重要程度    | Critical / High / Medium / Low |
| **测试类型** | 测什么方面   | 功能测试 / 安全性测试 / 性能测试            |
| **自动化**  | 是否自动执行  | 是 / 否                          |
| **操作步骤** | 具体怎么测   | 1. xxx<br>2. xxx<br>3. xxx     |
| **预期结果** | 应该发生什么  | 状态码 200，显示成功                   |
| **实际结果** | 实际发生了什么 | 状态码 200 ✓                      |
| **状态**   | 通过还是失败  | PASS / FAIL                    |

### 1.3 优先级详解

```
Critical（严重） - 必须测试
  → 核心功能，如登录、支付、数据保存

High（高） - 很重要
  → 主要功能，如注册、搜索、下单

Medium（中） - 一般重要
  → 辅助功能，如修改头像、查看评论

Low（低） - 可有可无
  → 锦上添花的功能，如界面动画、快捷键
```

**记忆技巧：**

- Critical = 挂了会死人的（比如银行转账）
- High = 挂了会很麻烦的（比如登录）
- Medium = 挂了影响不大的（比如修改昵称）
- Low = 挂了无所谓的（比如背景颜色）

## 三、测试类型有哪些？

### 3.1 功能测试

**目的：** 验证功能是否能正常工作

**例子：**

- 用户能否成功注册
- 购物车能否添加商品
- 订单能否提交成功

**在项目中：**

```
Test/test_auth.py      # 认证功能测试
Test/test_breed.py     # 品种管理功能测试
Test/test_charts.py    # 图表功能测试
```

### 3.2 安全性测试

**目的：** 检查系统是否安全

**例子：**

- SQL 注入攻击（试图绕过登录）
- XSS 攻击（注入恶意脚本）
- 权限控制（未登录能否访问后台）

**在项目中：**

```
Test/test_security.py  # 完整的安全性测试
```

**经典案例（来自 test_auth.py）：**

```python
def test_sql_injection_login(self, client, db):
    """SQL 注入攻击 - 登录接口"""

    # 尝试用 SQL 注入绕过密码验证
    payload = "' OR '1'='1"  # 这是一个经典的 SQL 注入语句

    response = client.post('/login', data={
        'username': payload,  # 用注入语句当用户名
        'password': 'anything'  # 随便输个密码
    })

    # 期望：应该拒绝登录
    assert '用户名或密码错误' in response.text
```

**通俗解释：**
黑客试图用特殊字符欺骗数据库，让它以为输入的是正确的用户名和密码。安全性测试就是要防止这种攻击。

### 3.3 性能测试

**目的：** 检查系统快不快、稳不稳定

**例子：**

- 1000 人同时访问，页面会不会卡
- 连续使用 1 小时，内存会不会爆
- 数据库查询会不会太慢

**在项目中：**

```
Test/test_performance.py  # 性能测试
```

### 3.4 集成测试

**目的：** 检查多个模块配合得好不好

**例子：**

- 用户下单后，库存会不会自动减少
- 支付成功后，订单状态会不会自动更新

**在项目中：**

```
Test/test_integration.py  # 集成测试
```

### 4.1 找到最简单的测试文件


### 4.2 阅读代码

```python
"""
图表功能测试用例
"""
from .test_framework import test_case, test_manager, TestResult


class TestChartsFunctionality:
    """图表功能测试类"""

    @test_case('TC-CHART-001', priority='Medium', expected='图表页面正常加载')
    def test_chart_page_loads(self, client):
        """测试图表页面加载"""
        result = TestResult('TC-CHART-001', 'test_chart_page_loads', '图表功能', 'Medium')
        result.expected_result = '图表页面正常加载，状态码 200'

        try:
            # 访问图表页面
            response = client.get('/chart')

            # 验证状态码
            assert response.status_code == 200

            # 验证页面包含关键字
            response_text = response.get_data(as_text=True)
            assert '图表' in response_text or 'Chart' in response_text

            result.status = 'PASS'
            result.actual_result = '图表页面加载成功'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'页面加载失败：{str(e)}'
            raise
        finally:
            test_manager.record_result(result)
```

### 4.3 代码解析

**逐行解释：**

```python
# 第 1 行：文档字符串，说明这个文件是干什么的
"""
图表功能测试用例
"""

# 第 2 行：导入测试框架的工具
from .test_framework import test_case, test_manager, TestResult
# test_case    → 装饰器，用来标记这是一个测试用例
# test_manager → 管理器，用来记录测试结果
# TestResult   → 测试结果类，用来创建结果对象

# 第 3 行：定义一个测试类
class TestChartsFunctionality:
    """图表功能测试类"""
    # 类名通常以 Test 开头，表示这是一个测试类

    # 第 4 行：定义测试方法
    @test_case('TC-CHART-001', priority='Medium', expected='图表页面正常加载')
    def test_chart_page_loads(self, client):
        """测试图表页面加载"""
        # @test_case 装饰器告诉框架：这是一个测试用例
        # 'TC-CHART-001' → 测试用例编号
        # priority='Medium' → 优先级中等
        # expected='...' → 预期结果描述

        # 第 5 行：创建测试结果对象
        result = TestResult('TC-CHART-001', 'test_chart_page_loads', '图表功能', 'Medium')
        result.expected_result = '图表页面正常加载，状态码 200'
        # 这个对象用来记录测试的结果

        try:
            # 第 6 行：执行实际操作
            response = client.get('/chart')
            # client → 模拟浏览器
            # .get('/chart') → 访问 /chart 这个网址

            # 第 7 行：断言（验证结果）
            assert response.status_code == 200
            # assert → 如果后面条件为真，继续执行；如果为假，抛出异常
            # status_code == 200 → HTTP 状态码 200 表示成功

            # 第 8 行：进一步验证
            response_text = response.get_data(as_text=True)
            assert '图表' in response_text or 'Chart' in response_text
            # 获取页面内容，检查是否包含"图表"这个词

            # 第 9 行：记录成功
            result.status = 'PASS'
            result.actual_result = '图表页面加载成功'

        except AssertionError as e:
            # 第 10 行：如果测试失败
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'页面加载失败：{str(e)}'
            raise  # 重新抛出异常，让 pytest 知道测试失败了

        finally:
            # 第 11 行：无论成功失败，都要执行
            test_manager.record_result(result)
            # 把测试结果记录下来，生成报告时会用到
```

## 六、知识总结

### 今日重点：

1. ✅ 测试用例 = 考试题，有编号、步骤、预期结果
2. ✅ 优先级分四级：Critical > High > Medium > Low
3. ✅ 测试类型：功能、安全、性能、集成
4. ✅ 会用 pytest 运行简单测试

### 关键术语：

- **Test Case（测试用例）**：一道测试题目
- **Priority（优先级）**：重要程度
- **Assertion（断言）**：验证结果是否符合预期
- **PASS/FAIL**：通过/失败

### 常见问题：

**Q1: 为什么有些测试用例标 Critical，有些标 Medium？**

> A: 看功能的重要性。登录功能是 Critical，因为用户登不上就什么都做不了；修改头像是 Medium，因为头像不好看不会影响核心功能。

**Q2: assert 失败了会怎样？**

> A: pytest 会捕获这个异常，标记测试为 FAILED，然后在报告中显示错误信息。

**Q3: 为什么要用 try-finally？**

> A: finally 里的代码无论如何都会执行，确保测试结果被记录下来。

## 二、测试代码的命名规则

### 2.1 文件命名规则

**规则：** `test_功能模块.py`

**例子：**

```
✅ test_auth.py          # 认证模块测试
✅ test_breed.py         # 品种管理测试
✅ test_charts.py        # 图表功能测试
❌ AuthTest.py           # 错误：不符合命名规范
❌ test.py               # 错误：没有体现功能模块
```

**为什么这样命名？**

- pytest 会自动识别 `test_` 开头的文件
- 看到文件名就知道是测什么的
- 方便管理和查找

**小技巧：**

```bash
# 列出所有测试文件
dir /b Test\test_*.py
```

### 2.2 类的命名规则

**规则：** `Test + 功能模块名 + Functionality/Tests`

**例子：**

```python
# ✅ 正确的命名
class TestUserAuthentication:      # 用户认证测试类
class TestBreedManagement:         # 品种管理测试类
class TestChartsFunctionality:     # 图表功能测试类

# ❌ 错误的命名
class AuthTest:                    # 不符合规范
class Test:                        # 太简单，不知道测什么
```

**记忆技巧：**

- 必须以 `Test` 开头（大写 T）
- 中间是功能模块名（驼峰命名）
- 可以加 `Functionality` 或 `Tests` 后缀

### 2.3 方法的命名规则

**规则：** `test_ + 具体测试场景`

**例子：**

```python
# ✅ 清晰的命名
def test_login_success(self):              # 登录成功
def test_register_duplicate_username(self) # 注册重复用户名
def test_sql_injection_attack(self)        # SQL 注入攻击

# ❌ 糟糕的命名
def test1(self):                           # 不知道测什么
def login(self):                           # 缺少 test_前缀
def testLogin(self):                       # 应该用下划线分隔
```

**最佳实践：**

- 用 `test_` 开头
- 全部小写
- 单词间用下划线 `_` 分隔
- 名字要能清楚表达测试场景

## 三、fixture 详解（重点！）

### 3.1 什么是 fixture？

**通俗解释：** fixture = **测试前的准备工作**

就像做饭前要买菜、洗菜一样，测试前也要做准备：

- 准备测试数据
- 创建数据库连接
- 模拟浏览器
- 创建测试用户

**官方定义：**

> Fixture 是 pytest 提供的一种依赖注入机制，用于在测试执行前设置测试环境。

### 3.2 常见的 fixture

**本项目中的 fixture：**

| Fixture 名称         | 作用         | 来源文件        |
| ------------------ | ---------- | ----------- |
| `client`           | 模拟浏览器      | conftest.py |
| `db`               | 数据库会话      | conftest.py |
| `logged_in_client` | 已登录的浏览器    | conftest.py |
| `app`              | Flask 应用实例 | conftest.py |

### 3.3 client fixture 详解

**作用：** 模拟浏览器访问网站

**不用 fixture 的写法（麻烦）：**

```python
from app import app

def test_login():
    # 每次都要手动创建客户端
    client = app.test_client()

    # 手动启动应用
    with app.app_context():
        response = client.post('/login', data={
            'username': 'test',
            'password': '123456'
        })

        assert response.status_code == 200
```

**使用 fixture 的写法（简单）：**

```python
def test_login(client):  # ← 直接用！
    """client 会自动创建"""
    response = client.post('/login', data={
        'username': 'test',
        'password': '123456'
    })

    assert response.status_code == 200
```

**对比：**

- 不用 fixture：10 行代码
- 使用 fixture：5 行代码
- **节省了 50% 的代码量！**

### 3.4 db fixture 详解

**作用：** 提供数据库会话，自动清理数据

**实际例子：**

```python
def test_user_registration(client, db):
    """
    client → 模拟浏览器
    db     → 数据库会话（自动创建和清理）
    """
    # 使用 db 创建用户
    user = User(username='testuser')
    user.set_password('password123')
    db.session.add(user)
    db.session.commit()

    # 测试结束后，db fixture 会自动清理数据
    # 不需要你手动删除！
```

**自动清理的好处：**

- ✅ 测试之间互不影响
- ✅ 不会污染数据库
- ✅ 每次测试都是干净的环境

### 3.5 fixture 在哪定义的？

**查看 conftest.py 文件：**

打开文件：`conftest.py`（在项目根目录）

你会看到类似这样的代码：

```python
import pytest
from app import create_app
from models import db as database

@pytest.fixture
def app():
    """创建 Flask 应用实例"""
    app = create_app()

    with app.app_context():
        database.create_all()  # 创建所有表
        yield app              # 返回给测试使用
        database.drop_all()    # 测试完后删除所有表


@pytest.fixture
def client(app):
    """创建测试客户端"""
    return app.test_client()


@pytest.fixture
def db(app):
    """创建数据库会话"""
    with app.app_context():
        yield database.session
```

**代码解析：**

```python
# 第 1 步：导入需要的模块
import pytest
from app import create_app
from models import db as database

# 第 2 步：定义 fixture
@pytest.fixture  # ← 这个装饰器告诉 pytest 这是一个 fixture
def app():
    """创建 Flask 应用实例"""
    # 准备阶段
    app = create_app()  # 创建应用

    with app.app_context():
        database.create_all()  # 创建数据库表

        # 返回给测试使用
        yield app  # ← yield 后面的代码测试完后执行

        # 清理阶段
        database.drop_all()  # 删除数据库表


@pytest.fixture
def client(app):
    """创建测试客户端"""
    # app 参数会自动调用上面的 app fixture
    return app.test_client()


@pytest.fixture
def db(app):
    """创建数据库会话"""
    with app.app_context():
        yield database.session  # 返回数据库会话对象
```

**yield 关键字的作用：**

```python
yield app  # 分成两部分理解：

# yield 之前：准备阶段（测试前执行）
- 创建应用
- 创建数据库表

# yield 之后：清理阶段（测试后执行）
- 删除数据库表
- 释放资源
```

### 3.6 如何使用 fixture？

**方法 1：作为函数参数（最常用）**

```python
def test_something(client, db):
    """
    参数名要和 fixture 名一致
    client → 使用 client fixture
    db     → 使用 db fixture
    """
    # 直接使用
    response = client.get('/some-page')
```

**方法 2：使用装饰器**

```python
@pytest.mark.usefixtures("client", "db")
def test_something():
    """
    用装饰器指定使用哪些 fixture
    """
    # 但这样不能直接使用，要用 request 参数
    pass
```

**推荐使用方法 1**，更简单直观！

## 四、装饰器 @test_case 详解

### 4.1 什么是装饰器？

**通俗解释：** 装饰器 = **包装器**

就像给礼物包装一样，装饰器给函数"包装"一些额外功能。

**生活中的例子：**

```
原始功能：开车
装饰器：导航系统
组合后：开车 + 导航 = 会导航的开车

原始功能：拍照
装饰器：美颜滤镜
组合后：拍照 + 美颜 = 会美颜的拍照
```

**代码中的例子：**

```python
# 原始函数
def login():
    print("登录功能")

# 加上日志装饰器
@log_decorator
def login():
    print("登录功能")
    # 现在会自动记录日志了！
```

### 4.2 @test_case 装饰器的作用

**作用：** 给测试函数添加元数据

**元数据包括：**

- 测试用例编号（如 TC-AUTH-001）
- 优先级（Critical/High/Medium/Low）
- 预期结果描述
- 测试类型

### 4.3 查看 @test_case 的实现

打开文件：`Test/test_framework.py`

找到 `test_case` 函数的定义：

```python
def test_case(case_id, priority='Medium', expected='', test_type='功能测试'):
    """
    测试用例装饰器

    参数:
        case_id: 测试用例编号，如 'TC-AUTH-001'
        priority: 优先级，默认 Medium
        expected: 预期结果描述
        test_type: 测试类型，默认功能测试
    """
    def decorator(func):
        # 把测试信息绑定到函数上
        func._test_case_id = case_id
        func._test_priority = priority
        func._test_expected = expected
        func._test_type = test_type

        # 返回原函数
        return func

    return decorator
```

**代码解析：**

```python
# 第 1 层：接收装饰器参数
def test_case(case_id, priority='Medium', expected='', test_type='功能测试'):
    # case_id='TC-AUTH-001'
    # priority='High'
    # expected='注册成功'
    # test_type='功能测试'

    # 第 2 层：真正的装饰器
    def decorator(func):
        # func 是被装饰的函数（如 test_register_success）

        # 给函数添加属性
        func._test_case_id = case_id       # 添加 ID
        func._test_priority = priority     # 添加优先级
        func._test_expected = expected     # 添加预期结果
        func._test_type = test_type        # 添加测试类型

        return func  # 返回原函数

    return decorator
```

**使用示例：**

```python
# 不使用装饰器
def test_register_success():
    pass

# 手动添加属性（麻烦）
test_register_success._test_case_id = 'TC-AUTH-001'
test_register_success._test_priority = 'High'

# 使用装饰器（简单）
@test_case('TC-AUTH-001', priority='High', expected='注册成功')
def test_register_success():
    pass
```

### 4.4 装饰器执行流程

**执行顺序：**

```python
# 代码执行顺序
1. Python 解释器遇到 @test_case(...)
2. 执行 test_case(...) 函数，返回 decorator
3. 用 decorator 包装 test_register_success 函数
4. 给函数添加属性：_test_case_id, _test_priority 等
5. 返回包装后的函数

# 测试执行时
pytest 收集测试 → 发现函数有 _test_case_id 属性 
→ 知道这是测试用例 → 执行测试 → 记录结果
```

## 五、完整的测试代码解析

### 5.1 逐行分析

让我们详细分析一个完整的测试：

```python
# ========== 导入部分 ==========
from .test_framework import test_case, test_manager, TestResult
# 从 test_framework 模块导入三个工具
# test_case    → 装饰器，标记测试用例
# test_manager → 管理器，记录测试结果
# TestResult   → 结果类，创建结果对象


# ========== 测试类定义 ==========
class TestUserAuthentication:
    """用户认证测试类"""
    # 类名：Test + UserAuthentication
    # 文档字符串：说明这个类测什么

    # ========== 测试方法定义 ==========
    @test_case('TC-AUTH-001', priority='High', expected='注册成功，跳转到登录页面')
    def test_register_success(self, client, db):
        """用户注册 - 成功场景"""

        # ========== 创建结果对象 ==========
        result = TestResult('TC-AUTH-001', 'test_register_success', '用户认证', 'High')
        result.expected_result = '注册成功，创建新用户'
        # TestResult(用例 ID, 方法名，模块名，优先级)
        # 创建结果对象，用来记录测试结果

        # ========== try-finally 结构 ==========
        try:
            # ====== 执行测试 ======

            # 1. 模拟浏览器发送 POST 请求
            response = client.post('/register', data={
                'username': 'testuser001',
                'password': 'TestPass123!'
            }, follow_redirects=True)
            # client.post() → 模拟提交表单
            # data={}       → 表单数据
            # follow_redirects=True → 自动跟随重定向

            # 2. 获取响应文本
            response_text = response.get_data(as_text=True)

            # 3. 断言验证
            assert response.status_code == 200
            # 状态码必须是 200（成功）

            assert '登录' in response_text
            # 页面必须包含"登录"二字（说明跳转到了登录页）

            # 4. 验证数据库
            user = User.query.filter_by(username='testuser001').first()
            assert user is not None
            # 数据库中必须有这个用户
            assert user.username == 'testuser001'
            # 用户名必须正确

            # ====== 记录成功 ======
            result.status = 'PASS'
            result.actual_result = f'注册成功，用户 ID: {user.id}'

        except AssertionError as e:
            # ====== 处理失败 ======
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'注册失败：{str(e)}'
            raise  # 重新抛出异常，让 pytest 知道测试失败了

        finally:
            # ====== 无论成败都要执行 ======
            test_manager.record_result(result)
            # 记录测试结果到报告
```

## 七、常见错误与解决方案

### 错误 1：找不到 fixture

**错误信息：**

```
fixture 'client' not found
```

**原因：**

- conftest.py 不在正确的位置
- conftest.py 里没有定义 client fixture

**解决方案：**

1. 检查 conftest.py 是否在 `Test/conftest.py` 或项目根目录
2. 确认里面有 `@pytest.fixture def client(): ...`

### 错误 2：装饰器参数错误

**错误信息：**

```
TypeError: test_case() missing required argument 'case_id'
```

**原因：** 忘记传 case_id 参数

**错误示例：**

```python
@test_case(priority='High')  # ❌ 缺少 case_id
def test_something():
    pass
```

**正确示例：**

```python
@test_case('TC-XXX-001', priority='High')  # ✅ 有 case_id
def test_something():
    pass
```

### 错误 3：断言总是失败

**错误信息：**

```
AssertionError: assert '登录' in '<html>...</html>'
```

**可能原因：**

1. 页面真的没有这个词
2. 编码问题导致中文乱码

**调试方法：**

```python
response_text = response.get_data(as_text=True)
print(response_text)  # 打印出来看看
assert '登录' in response_text
```

## 九、知识总结

### 今日重点：

1. ✅ 文件命名：`test_功能.py`
2. ✅ 类命名：`Test + 功能名`
3. ✅ 方法命名：`test_场景`
4. ✅ **fixture = 测试准备工作**
5. ✅ **装饰器 = 给函数添加元数据**
6. ✅ 完整的测试代码结构

### 核心概念：

- **Fixture**：pytest 的依赖注入，自动准备测试环境
- **@pytest.fixture**：定义 fixture 的装饰器
- **yield**：分隔准备阶段和清理阶段
- **@test_case**：自定义装饰器，添加测试元数据
- **try-finally**：确保测试结果被记录

### 难点解答：

**Q1: fixture 的执行顺序是怎样的？**

> A: pytest 会先执行所有 fixture 的 yield 前代码，然后执行测试，最后执行 fixture 的 yield 后代码。

**Q2: 多个 fixture 之间有依赖关系怎么办？**

> A: 在被依赖的 fixture 参数里写上依赖的 fixture 名。比如 `def client(app):` 表示 client 依赖 app。

**Q3: 为什么装饰器要返回 decorator 函数？**

> A: 因为装饰器需要接收参数（如 case_id、priority），所以要分两层：外层接收装饰器参数，内层接收被装饰的函数。

```python
class TestResult:
    """测试结果数据类"""

    def __init__(self, case_id, method_name, module_name, priority):
        """
        初始化测试结果

        参数:
            case_id: 测试用例编号，如 'TC-AUTH-001'

        self.module_name = module_name   # 模块名

        return {
            'ID': self.case_id,
            '方法名': self.method_name,
            '模块': self.module_name,
            '优先级': self.priority,
            '预期结果': self.expected_result,
            '实际结果': self.actual_result,
            '状态': self.status,
            '错误信息': self.error_message,
```

```python
class TestManager:
    """测试结果管理器"""

    def __init__(self):
        """初始化管理器"""
        self.results = []           # 存储所有测试结果
        self.start_time = None      # 测试开始时间
        self.end_time = None        # 测试结束时间
        self.tester_name = ''       # 测试人员姓名

    def record_result(self, result):
        """记录一个测试结果"""
        # 填充测试人员和日期
        result.tester = self.tester_name
        result.test_date = datetime.now().strftime('%Y-%m-%d %H:%M:%S')

        # 添加到结果列表
        self.results.append(result)

        # 打印简要信息
        status_symbol = '✓' if result.status == 'PASS' else '✗'
        print(f"{status_symbol} {result.case_id}: {result.method_name} - {result.status}")

    def get_summary(self):
        """获取测试总结"""
        total = len(self.results)
        passed = sum(1 for r in self.results if r.status == 'PASS')
        failed = sum(1 for r in self.results if r.status == 'FAIL')
        pending = sum(1 for r in self.results if r.status == 'PENDING')

        # 计算通过率
        pass_rate = (passed / total * 100) if total > 0 else 0

        return {
            'total': total,
            'passed': passed,
            'failed': failed,
            'pending': pending,
            'pass_rate': f'{pass_rate:.1f}%',
            'execution_time': self.get_total_execution_time()
        }

    def get_total_execution_time(self):
        """获取总执行时间"""
        total_time = sum(r.execution_time for r in self.results)
        return f'{total_time:.2f}秒'

    def export_to_txt(self, filename):
        """导出测试总结到 TXT 文件"""
        summary = self.get_summary()

        with open(filename, 'w', encoding='utf-8') as f:
            f.write('测试执行总结\n')
            f.write('=' * 50 + '\n\n')
            f.write(f'总计测试用例：{summary["total"]}\n')
            f.write(f'通过：{summary["passed"]}\n')
            f.write(f'失败：{summary["failed"]}\n')
            f.write(f'跳过：{summary["pending"]}\n')
            f.write(f'通过率：{summary["pass_rate"]}\n')
            f.write(f'总执行时间：{summary["execution_time"]}\n')

    def export_to_excel(self, filename):
        """导出详细测试结果到 Excel"""
        import pandas as pd

        # 将所有结果转换为字典列表
        data = [r.to_dict() for r in self.results]

        # 创建 DataFrame
        df = pd.DataFrame(data)

        # 导出到 Excel
        df.to_excel(filename, index=False, engine='openpyxl')

        print(f'✓ 测试结果已导出到：{filename}')


# 创建全局实例
test_manager = TestManager()
```

### 4.1 为什么要自定义装饰器？

**场景 2：添加通用逻辑**

比如想给每个测试自动计时：

```python
import time
from functools import wraps

def timer_decorator(func):
    """计算函数执行时间的装饰器"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        start_time = time.time()  # 记录开始时间
        result = func(*args, **kwargs)  # 执行原函数
        end_time = time.time()  # 记录结束时间
        execution_time = end_time - start_time
        print(f'{func.__name__} 执行了 {execution_time:.2f}秒')
        return result
    return wrapper

# 使用
@timer_decorator
def test_something():
    time.sleep(1)  # 模拟耗时操作

test_something()  # 输出：test_something 执行了 1.00 秒
```

### 4.2 从零开始写一个装饰器

**任务：** 写一个装饰器，给测试函数添加重试功能

**需求：**

- 如果测试失败，自动重试 3 次
- 3 次都失败才认为真的失败
- 每次重试间隔 1 秒

**步骤 1：分析需求**

```python
# 装饰器需要做什么？
1. 包装原函数
2. 执行原函数，如果成功直接返回
3. 如果失败，重试最多 3 次
4. 每次重试间隔 1 秒
5. 3 次都失败才抛出异常
```

**步骤 2：实现简单版本**

```python
import time
from functools import wraps

def retry(max_attempts=3, delay=1):
    """
    重试装饰器

    参数:
        max_attempts: 最大重试次数，默认 3 次
        delay: 重试间隔（秒），默认 1 秒
    """
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            attempts = 0

            while attempts < max_attempts:
                try:
                    # 尝试执行函数
                    return func(*args, **kwargs)

                except Exception as e:
                    attempts += 1
                    print(f'尝试 {attempts}/{max_attempts} 失败：{str(e)}')

                    if attempts >= max_attempts:
                        # 超过最大重试次数，重新抛出异常
                        raise

                    # 等待一段时间后重试
                    time.sleep(delay)

        return wrapper
    return decorator
```

**步骤 3：使用装饰器**

```python
@retry(max_attempts=3, delay=1)
def test_unreliable_feature():
    """测试不稳定的功能"""
    import random
    if random.random() < 0.7:  # 70% 概率失败
        raise AssertionError('网络超时')
    print('成功！')

# 运行测试
test_unreliable_feature()
```

### 4.3 结合 TestResult 的装饰器

**高级用法：** 自动记录执行时间

```python
import time
from functools import wraps
from test_framework import test_manager, TestResult

def auto_record_test(case_id, method_name, module_name, priority):
    """
    自动记录测试结果的装饰器

    不需要手动写 try-finally，装饰器会自动处理
    """
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            # 创建结果对象
            result = TestResult(case_id, method_name, module_name, priority)

            start_time = time.time()

            try:
                # 执行测试
                func(*args, **kwargs)

                # 成功
                result.status = 'PASS'
                result.actual_result = '测试通过'

            except AssertionError as e:
                # 失败
                result.status = 'FAIL'
                result.error_message = str(e)
                result.actual_result = f'测试失败：{str(e)}'
                raise

            finally:
                # 记录执行时间
                result.execution_time = time.time() - start_time

                # 自动记录结果
                test_manager.record_result(result)

        return wrapper
    return decorator
```

## 五、测试报告生成原理

### 5.1 HTML 报告是怎么来的？

**生成流程：**

```
1. pytest 收集所有测试用例
   ↓
2. 逐个执行测试
   ↓
3. 每个测试完成后调用 test_manager.record_result()
   ↓
4. 所有测试执行完毕
   ↓
5. 调用 report_generator.generate_html_report()
   ↓
6. 读取 test_manager.results 获取所有结果
   ↓
7. 生成 HTML 模板
   ↓
8. 写入文件 reports/report.html
```

## 四、断言（Assert）完全指南

### 4.1 什么是断言？

**通俗解释：** 断言 = **检查点**

就像老师检查作业一样，断言用来检查测试结果是否符合预期。

### 4.2 常用断言类型

#### 类型 1：状态码断言

```python
# 检查 HTTP 状态码
assert response.status_code == 200      # 成功
assert response.status_code == 201      # 创建成功
assert response.status_code == 302      # 重定向
assert response.status_code == 400      # 请求错误
assert response.status_code == 401      # 未授权
assert response.status_code == 403      # 禁止访问
assert response.status_code == 404      # 未找到
assert response.status_code == 500      # 服务器错误
```

**实际例子：**

```python
def test_create_user(client):
    # 创建用户应该返回 201
    response = client.post('/users', data={'name': 'test'})
    assert response.status_code == 201

def test_delete_user(client):
    # 删除不存在的用户应该返回 404
    response = client.delete('/users/999')
    assert response.status_code == 404
```

---

#### 类型 2：内容断言

## 五、边界情况处理

### 5.1 什么是边界情况？

**通俗解释：** 边界情况 = **极端情况**

就像考试，不仅要看中等难度的题，还要看最简单和最难的题。

### 5.2 常见的边界情况

#### 边界 1：空值

```python
# 测试空字符串
def test_empty_username(self, client, db):
    """用户名为空"""
    response = client.post('/register', data={
        'username': '',  # 空字符串
        'password': 'TestPass123'
    })
    assert response.status_code == 400  # 应该拒绝
```

#### 边界 2：最大值

```python
# 测试最大长度
def test_max_length_username(self, client, db):
    """用户名达到最大长度"""
    max_username = 'a' * 20  # 20 个字符
    response = client.post('/register', data={
        'username': max_username,
        'password': 'TestPass123'
    })
    assert response.status_code == 200  # 应该成功
```

#### 边界 3：超过最大值

```python
# 测试超过最大长度
def test_too_long_username(self, client, db):
    """用户名超过最大长度"""

        'username': too_long_username,
        'password': 'TestPass123'
    })

```

#### 边界 4：最小值

```python
# 测试最小长度
def test_min_length_username(self, client, db):

        'username': min_username,
```

#### 边界 5：低于最小值

```python

购物车功能测试

from .test_framework import test_case, test_manager, TestResult
from models import Product, CartItem
from test_tools.factories import UserFactory


class TestShoppingCartFunctionality:

    @test_case('TC-CART-001', priority='High', expected='成功添加商品')
    def test_add_product_to_cart_success(self, client, db, logged_in_client):
        """正常添加商品到购物车"""

                'quantity': 2

            cart_item = CartItem.query.filter_by(
                product_id=product.id
            ).first()
            assert cart_item is not None
            assert cart_item.quantity == 2

            result.status = 'PASS'
            result.actual_result = f'成功添加{cart_item.quantity}件商品'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            raise
        finally:
            test_manager.record_result(result)
```

## 二、SQL 注入测试

### 2.1 什么是 SQL 注入？

**通俗解释：** SQL 注入 = **欺骗数据库说假话**

就像有人冒充你说："我是你朋友，快把钱给我"一样，SQL 注入是黑客试图欺骗数据库执行恶意操作。

**实际例子：**

正常登录：

```python
username = "张三"
password = "123456"

# SQL 查询
SELECT * FROM users WHERE username='张三' AND password='123456'
```

SQL 注入攻击：

```python
username = "张三' OR '1'='1"  # 这是个陷阱！
password = "anything"

# SQL 查询变成了
SELECT * FROM users WHERE username='张三' OR '1'='1' AND password='anything'
#                                        ^^^^^^^^^^
#                                        永远为真！所以会返回所有用户！
```

### 2.2 常见的 SQL 注入 payload

**Payload 清单：**

| Payload                          | 作用         | 危险程度   |
| -------------------------------- | ---------- | ------ |
| `' OR '1'='1`                    | 绕过条件判断     | ⭐⭐⭐⭐⭐  |
| `admin' --`                      | 注释掉后面的密码验证 | ⭐⭐⭐⭐⭐  |
| `' OR 1=1 --`                    | 另一种绕过方式    | ⭐⭐⭐⭐   |
| `'; DROP TABLE users; --`        | 删除表（极度危险）  | ⭐⭐⭐⭐⭐+ |
| `' UNION SELECT * FROM users --` | 窃取其他表数据    | ⭐⭐⭐⭐⭐  |
| `1; WAITFOR DELAY '0:0:5'`       | 时间延迟攻击     | ⭐⭐⭐    |

**详细解释：**

#### Payload 1: `' OR '1'='1`

```python
# 原始 SQL
SELECT * FROM users WHERE username='{input}' AND password='{password}'

# 输入：' OR '1'='1
SELECT * FROM users WHERE username='' OR '1'='1' AND password='xxx'
#                              空字符串   OR 永远为真     任意密码
# 结果：因为'1'='1'永远为真，所以会返回所有用户！
```

#### Payload 2: `admin' --`

```python
# 原始 SQL
SELECT * FROM users WHERE username='{input}' AND password='{password}'

# 输入：admin' --
SELECT * FROM users WHERE username='admin' --' AND password='xxx'
#                                         ^^
#                                         注释符号！后面的都被忽略了
# 结果：只需要知道用户名是 admin，不需要密码！
```

#### Payload 3: `'; DROP TABLE users; --`

```python
# 如果系统执行多条 SQL 语句
# 输入这个会删除整个 users 表！
# 这就是为什么叫"极度危险"
```

### 2.3 如何测试 SQL 注入

**实战代码：**

```python
"""
安全性测试 - SQL 注入
"""
from .test_framework import test_case, test_manager, TestResult
from models import User


class TestSQLInjectionSecurity:
    """SQL 注入安全测试类"""

    @test_case('TC-SEC-001', priority='Critical', expected='SQL 注入攻击失败')
    def test_sql_injection_login_bypass(self, client, db):
        """SQL 注入攻击 - 绕过登录"""
        result = TestResult('TC-SEC-001', 'test_sql_injection_login_bypass', '安全性', 'Critical')
        result.expected_result = 'SQL 注入攻击被阻止，无法绕过登录'

        try:
            # 1. 先创建一个管理员用户
            admin = User(username='admin')
            admin.set_password('admin123')
            db.session.add(admin)
            db.session.commit()

            # 2. SQL 注入 payloads
            payloads = [
                "' OR '1'='1",           # 经典绕过
                "admin' --",              # 注释密码
                "' OR 1=1 --",            # 数字版绕过
                "' OR 'x'='x",            # 字母版绕过
                "admin' OR '1'='1",       # 组合攻击
            ]

            # 3. 逐个测试
            for payload in payloads:
                response = client.post('/login', data={
                    'username': payload,
                    'password': 'wrong_password'  # 故意用错密码
                }, follow_redirects=True)

                response_text = response.get_data(as_text=True)

                # 4. 验证：不应该登录成功
                assert '登录成功' not in response_text, \
                    f"SQL 注入漏洞！payload '{payload}' 绕过了登录验证！"

                # 应该提示错误
                assert '用户名或密码错误' in response_text or \
                       '无效' in response_text or \
                       response.status_code != 200

            result.status = 'PASS'
            result.actual_result = f'成功阻止 {len(payloads)} 种 SQL 注入攻击'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'发现 SQL 注入漏洞：{str(e)}'

            # 记录严重 Bug
            from bug_tracker import report_bug
            report_bug(
                title="CRITICAL: SQL 注入漏洞 - 可绕过登录验证",
                description=f"发现 SQL 注入漏洞，可使用 payload: {payload} 绕过登录",
                severity="Critical",
                priority="High",
                module="安全性 - 认证",
                steps_to_reproduce=[
                    "访问 /login 页面",
                    f"用户名输入：{payload}",
                    "密码输入任意值",
                    "点击登录",
                    "观察到无需正确密码即可登录"
                ],
                expected_result="应始终验证失败",
                actual_result=result.error_message
            )
            raise

        finally:
            test_manager.record_result(result)

    @test_case('TC-SEC-002', priority='Critical', expected='无法通过注入获取数据')
    def test_union_based_injection(self, client, db):
        """UNION 注入攻击测试"""
        result = TestResult('TC-SEC-002', 'test_union_based_injection', '安全性', 'Critical')
        result.expected_result = 'UNION 注入攻击被阻止'

        try:
            # UNION 注入 payload
            payload = "' UNION SELECT 1,2,3,4,5 --"

            response = client.get(f'/users?search={payload}')

            # 不应该返回异常数据
            response_text = response.get_data(as_text=True)

            # 验证没有泄露敏感信息
            assert 'password' not in response_text.lower()
            assert 'secret' not in response_text.lower()

            result.status = 'PASS'
            result.actual_result = 'UNION 注入攻击被阻止'

        except Exception as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            raise
        finally:
            test_manager.record_result(result)
```

### 2.4 防御 SQL 注入的方法

**方法 1：参数化查询（推荐）**

```python
# ❌ 错误写法（拼接 SQL）
sql = f"SELECT * FROM users WHERE username='{username}'"

# ✅ 正确写法（参数化）
sql = "SELECT * FROM users WHERE username=?"
cursor.execute(sql, (username,))
```

**方法 2：ORM（如 SQLAlchemy）**

```python
# ORM 自动处理参数化
user = User.query.filter_by(username=username).first()
# 底层会自动转义特殊字符
```

**方法 3：输入验证**

```python
# 只允许字母数字
if not re.match(r'^[a-zA-Z0-9_]+$', username):
    raise ValueError("用户名包含非法字符")
```

## 三、XSS（跨站脚本）攻击测试

### 3.1 什么是 XSS？

**通俗解释：** XSS = **在别人的网站里插入自己的坏代码**

就像有人在公告栏上贴了一张带病毒的传单，其他人看到就会被感染。

**实际例子：**

正常评论：

```
这个商品很好用！★★★★★
```

XSS 攻击评论：

```html
这个商品很好用！★★★★★
<script>alert('你的 Cookie 是：' + document.cookie)</script>
```

### 3.2 常见的 XSS payload

**Payload 清单：**

| Payload                            | 作用        | 危险程度  |
| ---------------------------------- | --------- | ----- |
| `<script>alert('XSS')</script>`    | 测试是否有 XSS | ⭐⭐    |
| `<img src=x onerror=alert('XSS')>` | 图片标签注入    | ⭐⭐⭐   |
| `<svg onload=alert('XSS')>`        | SVG 注入    | ⭐⭐⭐   |
| `javascript:alert('XSS')`          | JS 协议注入   | ⭐⭐⭐⭐  |
| `<iframe src="http://evil.com">`   | 嵌入恶意网站    | ⭐⭐⭐⭐⭐ |

### 3.3 如何测试 XSS

**实战代码：**

```python
"""
安全性测试 - XSS 攻击
"""
from .test_framework import test_case, test_manager, TestResult


class TestXSSSecurity:
    """XSS 攻击安全测试类"""

    @test_case('TC-SEC-003', priority='High', expected='XSS 脚本被过滤')
    def test_xss_in_comment(self, client, db, logged_in_client):
        """评论区 XSS 注入测试"""
        result = TestResult('TC-SEC-003', 'test_xss_in_comment', '安全性', 'High')
        result.expected_result = 'XSS 脚本被过滤或转义'

        try:
            # XSS payloads
            xss_payloads = [
                "<script>alert('XSS')</script>",
                "<img src=x onerror=alert('XSS')>",
                "<svg onload=alert('XSS')>",
                "javascript:alert('XSS')",
                "<body onload=alert('XSS')>"
            ]

            for payload in xss_payloads:
                # 尝试提交包含 XSS 的评论
                response = logged_in_client.post('/comment/add', data={
                    'content': f'这个商品不错！{payload}',
                    'rating': 5
                }, follow_redirects=True)

                # 1. 要么拒绝提交
                if response.status_code == 200:
                    response_text = response.get_data(as_text=True)

                    # 2. 要么过滤/转义了脚本
                    # 检查是否转义了 HTML 特殊字符
                    assert '<script>' in response_text or \
                           '<script>' not in response_text, \
                           "XSS 漏洞！脚本未被转义！"

            result.status = 'PASS'
            result.actual_result = f'成功阻止 {len(xss_payloads)} 种 XSS 攻击'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = f"发现 XSS 漏洞：{str(e)}"
            raise
        finally:
            test_manager.record_result(result)

    @test_case('TC-SEC-004', priority='Medium', expected='用户名中的脚本被转义')
    def test_xss_in_username(self, client, db):
        """用户名 XSS 注入测试"""
        result = TestResult('TC-SEC-004', 'test_xss_in_username', '安全性', 'Medium')
        result.expected_result = '用户名中的脚本被转义'

        try:
            # 注册包含脚本的用户名
            malicious_username = "<script>alert('XSS')</script>"

            response = client.post('/register', data={
                'username': malicious_username,
                'password': 'TestPass123'
            }, follow_redirects=True)

            response_text = response.get_data(as_text=True)

            # 验证：要么拒绝注册，要么转义了脚本
            if '注册成功' in response_text:
                # 检查显示时是否转义
                assert '<script>' in response_text or \
                       '<script>' not in response_text

            result.status = 'PASS'
            result.actual_result = '用户名 XSS 被正确转义'

        except Exception as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            raise
        finally:
            test_manager.record_result(result)
```

### 3.4 防御 XSS 的方法

**方法 1：HTML 转义（最常用）**

```python
from markupsafe import escape

# 转义后
escape("<script>alert('XSS')</script>")
# 输出：<script>alert(&#39;XSS&#39;)</script>

# 浏览器会显示：<script>alert('XSS')</script>（不会执行）
```

**方法 2：白名单过滤**

```python
# 只允许安全的 HTML 标签
allowed_tags = ['p', 'br', 'strong', 'em']
clean_html = bleach.clean(user_input, tags=allowed_tags)
```

**方法 3：设置 Content-Security-Policy**

```python
# Flask 中设置 CSP 头
@app.after_request
def set_csp(response):
    response.headers['Content-Security-Policy'] = "default-src 'self'"
    return response
```

## 四、权限控制测试

### 4.1 什么是权限控制？

**通俗解释：** 权限控制 = **不同的人做不同的事**

就像公司里：

- 员工只能看自己的文件
- 经理可以看部门的文件
- CEO 可以看所有文件

### 4.2 常见的权限问题

**问题场景：**

| 场景             | 描述        | 危险程度  |
| -------------- | --------- | ----- |
| 未登录访问          | 没登录就能访问后台 | ⭐⭐⭐⭐⭐ |
| 普通用户访问管理员页面    | 权限提升      | ⭐⭐⭐⭐⭐ |
| A 用户访问 B 用户的数据 | 越权访问      | ⭐⭐⭐⭐  |
| 已登出用户仍能访问      | 会话未失效     | ⭐⭐⭐⭐  |

### 4.3 如何测试权限控制

**实战代码：**

```python
"""
安全性测试 - 权限控制
"""
from .test_framework import test_case, test_manager, TestResult
from models import User


class TestAuthorizationSecurity:
    """权限控制安全测试类"""

    @test_case('TC-AUTH-001', priority='Critical', expected='未登录用户被重定向')
    def test_unauthorized_access_admin_page(self, client):
        """未登录用户访问管理页面"""
        result = TestResult('TC-AUTH-001', 'test_unauthorized_access_admin_page', '安全性', 'Critical')
        result.expected_result = '未登录用户被重定向到登录页'

        try:
            # 尝试访问管理页面
            response = client.get('/admin/dashboard', follow_redirects=False)

            # 应该重定向（302）或拒绝（401/403）
            assert response.status_code in [302, 401, 403], \
                f"未登录用户竟能访问管理页面！状态码：{response.status_code}"

            # 如果是重定向，应该指向登录页
            if response.status_code == 302:
                location = response.headers.get('Location', '')
                assert 'login' in location.lower() or 'auth' in location.lower()

            result.status = 'PASS'
            result.actual_result = f'正确拦截未登录访问（状态码：{response.status_code}）'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            result.actual_result = f'权限控制失效：{str(e)}'
            raise
        finally:
            test_manager.record_result(result)

    @test_case('TC-AUTH-002', priority='Critical', expected='普通用户无法访问管理员功能')
    def test_normal_user_cannot_access_admin_function(self, client, db):
        """普通用户尝试访问管理员功能"""
        result = TestResult('TC-AUTH-002', 'test_normal_user_cannot_access_admin_function', '安全性', 'Critical')
        result.expected_result = '普通用户被拒绝访问管理员功能'

        try:
            # 1. 创建普通用户
            normal_user = User(username='normal_user', is_admin=False)
            normal_user.set_password('password123')
            db.session.add(normal_user)
            db.session.commit()

            # 2. 以普通用户登录
            login_response = client.post('/login', data={
                'username': 'normal_user',
                'password': 'password123'
            }, follow_redirects=True)
            assert login_response.status_code == 200

            # 3. 尝试访问管理员功能
            admin_response = client.get('/admin/users/delete/1', follow_redirects=True)

            # 应该被拒绝
            assert admin_response.status_code in [403, 401], \
                f"普通用户竟能访问管理员功能！状态码：{admin_response.status_code}"

            result.status = 'PASS'
            result.actual_result = '正确拒绝普通用户的管理员权限'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = f"权限提升漏洞：{str(e)}"
            raise
        finally:
            test_manager.record_result(result)

    @test_case('TC-AUTH-003', priority='High', expected='用户只能访问自己的数据')
    def test_user_cannot_access_other_user_data(self, client, db):
        """用户尝试访问其他用户的数据"""
        result = TestResult('TC-AUTH-003', 'test_user_cannot_access_other_user_data', '安全性', 'High')
        result.expected_result = '用户无法访问其他用户的数据'

        try:
            # 1. 创建两个用户
            user_a = User(username='user_a', email='a@example.com')
            user_a.set_password('password123')
            db.session.add(user_a)

            user_b = User(username='user_b', email='b@example.com')
            user_b.set_password('password123')
            db.session.add(user_b)
            db.session.commit()

            user_b_id = user_b.id

            # 2. user_a 登录
            client.post('/login', data={
                'username': 'user_a',
                'password': 'password123'
            })

            # 3. user_a 尝试访问 user_b 的资料
            response = client.get(f'/user/profile/{user_b_id}', follow_redirects=True)

            # 应该被拒绝或显示错误
            assert response.status_code in [403, 404] or \
                   '无权访问' in response.get_data(as_text=True) or \
                   '找不到' in response.get_data(as_text=True)

            result.status = 'PASS'
            result.actual_result = '正确防止越权访问'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = f"越权访问漏洞：{str(e)}"
            raise
        finally:
            test_manager.record_result(result)
```

## 五、性能测试基础

### 5.1 什么是性能测试？

**通俗解释：** 性能测试 = **测试系统能抗多少人同时使用**

就像测试一座桥能通过多少辆车：

- 1 辆车通过 → 没问题 ✓
- 10 辆车同时通过 → 没问题 ✓
- 100 辆车同时通过 → 有点晃 ✓
- 1000 辆车同时通过 → 塌了！❌

### 5.2 性能测试的类型

| 类型        | 目的          | 例子           |
| --------- | ----------- | ------------ |
| **压力测试**  | 测试极限在哪里     | 不断增加用户直到系统崩溃 |
| **负载测试**  | 测试正常负载下的表现  | 模拟 100 人同时访问 |
| **稳定性测试** | 测试长时间运行是否稳定 | 持续运行 24 小时   |
| **并发测试**  | 测试同时访问的表现   | 1000 人在同一秒点击 |

### 5.3 性能测试的关键指标

**指标说明：**

| 指标          | 含义            | 合理范围  |
| ----------- | ------------- | ----- |
| **响应时间**    | 从请求到收到响应的时间   | < 2 秒 |
| **吞吐量**     | 每秒处理的请求数（QPS） | 越高越好  |
| **错误率**     | 失败的请求比例       | < 1%  |
| **CPU 使用率** | CPU 占用百分比     | < 80% |
| **内存使用率**   | 内存占用百分比       | < 80% |

### 5.4 简单的性能测试代码

**实战代码：**

```python
"""
性能测试基础
"""
import time
import statistics
from .test_framework import test_case, test_manager, TestResult


class TestPerformanceBasics:
    """性能基础测试类"""

    @test_case('TC-PERF-001', priority='Medium', expected='首页加载时间小于 2 秒')
    def test_homepage_load_time(self, client):
        """测试首页加载时间"""
        result = TestResult('TC-PERF-001', 'test_homepage_load_time', '性能测试', 'Medium')
        result.expected_result = '首页加载时间小于 2 秒'

        try:
            # 测试 10 次取平均值
            load_times = []

            for i in range(10):
                start_time = time.time()

                response = client.get('/')

                end_time = time.time()
                load_time = end_time - start_time
                load_times.append(load_time)

                # 验证状态码
                assert response.status_code == 200

            # 计算统计数据
            avg_time = statistics.mean(load_times)
            min_time = min(load_times)
            max_time = max(load_times)

            # 平均加载时间应该小于 2 秒
            assert avg_time < 2.0, f"首页平均加载时间为{avg_time:.2f}秒，超过 2 秒"

            result.status = 'PASS'
            result.actual_result = f'首页平均加载{avg_time:.2f}秒（最快{min_time:.2f}s，最慢{max_time:.2f}s）'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            raise
        finally:
            test_manager.record_result(result)

    @test_case('TC-PERF-002', priority='High', expected='搜索接口响应时间小于 1 秒')
```

## 二、什么是 API 接口测试？

### 2.1 通俗解释

**API 接口 = 服务员**

就像你去餐厅点菜：

- 你（前端/客户端）告诉服务员（API）要什么菜
- 服务员去厨房（后端/数据库）取菜
- 服务员把菜端给你（返回响应）

**接口测试 = 检查服务员是否称职**

需要检查：

- 点菜是否正确传达 ✓
- 上菜速度是否够快 ✓
- 菜品是否正确 ✓
- 服务态度是否好 ✓

### 2.2 为什么需要接口测试？

**实际例子：**

假设有一个用户注册接口：POST /api/users/register

**需要测试：**

| 测试点    | 说明         | 重要性   |
| ------ | ---------- | ----- |
| 正常注册   | 输入合法的用户名密码 | ⭐⭐⭐⭐⭐ |
| 用户名重复  | 使用已存在的用户名  | ⭐⭐⭐⭐⭐ |
| 密码太短   | 密码少于 6 位   | ⭐⭐⭐⭐  |
| 缺少必填字段 | 不提供用户名或密码  | ⭐⭐⭐⭐  |
| 特殊字符   | 用户名包含@#$%等 | ⭐⭐⭐   |
| 超长输入   | 用户名超过最大长度  | ⭐⭐⭐   |

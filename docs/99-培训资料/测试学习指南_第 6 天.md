## 二、什么是 API 接口测试？

### 2.1 通俗解释

**API 接口 = 服务员**

就像你去餐厅点菜：
- 你（前端/客户端）告诉服务员（API）要什么菜
- 服务员去厨房（后端/数据库）取菜
- 服务员把菜端给你（返回响应）

**接口测试 = 检查服务员是否称职**

需要检查：
- 点菜是否正确传达 ✓
- 上菜速度是否够快 ✓
- 菜品是否正确 ✓
- 服务态度是否好 ✓

### 2.2 为什么需要接口测试？

假设有一个用户注册接口：POST /api/users/register

**需要测试的场景：**

| 测试点 | 说明 | 重要性 |
|--------|------|--------|
| 正常注册 | 输入合法的用户名密码 | ⭐⭐⭐⭐⭐ |
| 用户名重复 | 使用已存在的用户名 | ⭐⭐⭐⭐⭐ |
| 密码太短 | 密码少于 6 位 | ⭐⭐⭐⭐ |
| 缺少必填字段 | 不提供用户名或密码 | ⭐⭐⭐⭐ |
| 特殊字符 | 用户名包含@#$%等 | ⭐⭐⭐ |
| 超长输入 | 用户名超过最大长度 | ⭐⭐⭐ |

## 三、RESTful API 基础

### 3.1 常见的 HTTP 方法

| 方法 | 作用 | 例子 | 返回码 |
|------|------|------|--------|
| GET | 获取资源 | 获取用户列表 | 200 OK |
| POST | 创建资源 | 创建新用户 | 201 Created |
| PUT | 更新资源（全量） | 修改用户信息 | 200 OK |
| PATCH | 更新资源（部分） | 只修改邮箱 | 200 OK |
| DELETE | 删除资源 | 删除用户 | 204 No Content |

### 3.2 标准的 API 响应格式

成功的响应：
```json
{
    "code": 200,
    "message": "success",
    "data": {
        "user_id": 1,
        "username": "张三",
        "email": "zhangsan@example.com"
    }
}
```

失败的响应：
```json
{
    "code": 400,
    "message": "用户名已存在",
    "data": null
}
```

### 3.3 常见的状态码

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 200 | OK | 请求成功 |
| 201 | Created | 创建成功 |
| 204 | No Content | 删除成功（无返回） |
| 400 | Bad Request | 请求参数错误 |
| 401 | Unauthorized | 未登录 |
| 403 | Forbidden | 无权限 |
| 404 | Not Found | 资源不存在 |
| 500 | Internal Server Error | 服务器错误 |

### 4.1 测试 GET 接口 - 获取用户列表

```python
"""
接口测试 - 用户管理模块
"""
from .test_framework import test_case, test_manager, TestResult


class TestUserAPI:
    """用户接口测试类"""

    @test_case('TC-API-001', priority='High', expected='返回用户列表')
    def test_get_users_success(self, client, db):
        """获取用户列表 - 成功场景"""
        result = TestResult('TC-API-001', 'test_get_users_success', '用户接口', 'High')
        result.expected_result = '返回 200 状态码和用户列表'

        try:
            # 1. 准备测试数据 - 创建 3 个用户
            from models import User
            for i in range(3):
                user = User(username=f'test_user_{i}')
                user.set_password('password123')
                db.session.add(user)
            db.session.commit()

            # 2. 发送 GET 请求
            response = client.get('/api/users')

            # 3. 验证状态码
            assert response.status_code == 200, f"期望 200，实际{response.status_code}"

            # 4. 验证返回的是 JSON
            assert 'application/json' in response.content_type

            # 5. 解析 JSON 数据
            json_data = response.get_json()

            # 6. 验证数据结构
            assert 'data' in json_data or 'users' in json_data or isinstance(json_data, list)

            # 7. 验证用户数量
            if isinstance(json_data, list):
                assert len(json_data) == 3
            elif 'data' in json_data:
                assert len(json_data['data']) == 3

            result.status = 'PASS'
            result.actual_result = f'成功返回{len(json_data)}个用户'

        except AssertionError as e:
            result.status = 'FAIL'
            result.error_message = str(e)
            raise
        finally:
            test_manager.record_result(result)
```

## 九、知识总结

### 今日重点：

1. **GET 接口测试** - 验证数据获取、状态码、JSON 结构
2. **POST 接口测试** - 验证创建资源、参数验证、错误处理
3. **PUT 接口测试** - 验证更新操作、部分更新、完整性检查
4. **DELETE 接口测试** - 验证删除操作、级联删除、重复删除
5. **JSON 验证** - 验证字段、类型、结构、Schema
6. **参数化测试** - 提高测试效率、减少重复代码
7. **认证测试** - 验证接口的访问控制

### 核心概念：

- **状态码断言** - 200/201/204 表示成功，4xx 表示客户端错误，5xx 表示服务器错误
- **JSON 响应验证** - 检查字段存在性、类型、值范围、嵌套结构
- **参数化测试** - 一次编写，多次执行，覆盖多种场景
- **Fixture 复用** - client、db、logged_in_client 等 fixture 的使用

### 难点解答：

**Q1: 如何验证复杂的 JSON 嵌套结构？**

答：分层验证，先验证顶层字段，再逐层深入；或者使用 JSON Schema 库进行完整验证。

**Q2: 参数化测试和普通测试有什么区别？**

答：参数化测试会自动生成多个测试用例，每个参数组合都会独立执行一次，适合测试边界值和多种输入场景。

**Q3: 如何处理异步接口的测试？**

答：可以使用 Python 的 asyncio 库，或者在 Flask 中使用测试客户端的同步版本。对于真正的异步框架（如 FastAPI），需要使用 httpx 等异步 HTTP 客户端。

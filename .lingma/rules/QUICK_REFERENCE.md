# 项目管理快速参考

> 本文档提供项目管理的快速参考指南，方便日常查阅。

## 📋 长任务清单模板

```markdown
## 任务：[任务名称]

**开始时间**：YYYY-MM-DD  
**优先级**：高/中/低

### 任务分解

- [ ] 步骤 1：[描述]
- [ ] 步骤 2：[描述]
- [ ] 步骤 3：[描述]

### 当前进度

- 已完成：X/Y
- 正在进行：步骤 N
```

**使用要点**：
- ✅ 复杂任务必须创建清单
- ✅ 每完成一步立即汇报
- ✅ 中断时保存状态
- ✅ 完成后提供总结

## 📝 Git 提交规范

### Commit Message 格式

```bash
# 标准格式
git commit -m "feat(auth): 添加 JWT 认证"

# 简化格式（推荐）
git commit -m "v1.0.1"
git commit -m "v1.1.0"
```

### Type 类型

- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `refactor`: 重构
- `test`: 测试相关

### 版本号规则

- `v1.0.0` → `v1.0.1`: Bug 修复（PATCH）
- `v1.0.0` → `v1.1.0`: 新功能（MINOR）
- `v1.0.0` → `v2.0.0`: 重大变更（MAJOR）

## 📚 文档维护清单

### 功能开发完成后

必须更新：
- [ ] 功能文档（docs/04-特性文档/）
- [ ] CHANGELOG.md
- [ ] API 文档（如有变更）
- [ ] README.md（如影响用户）

### Bug 修复后

必须记录：
- [ ] Bug 描述和原因
- [ ] 解决方案
- [ ] 回归测试
- [ ] 预防措施

### 文档质量标准

✅ **必须包含**：
- 清晰的概述
- 技术实现细节
- 代码示例
- 使用示例
- 更新日期

❌ **避免**：
- 过时的内容
- 无解释的代码
- 冗长无重点

---

## 🧪 测试要求

### 覆盖率标准

- 新功能：≥ 80%
- 核心逻辑：≥ 95%
- API 接口：100%

### 测试命令

```bash
# 运行所有测试
pytest Test/ -v

# 检查覆盖率
pytest --cov=. --cov-report=html

# 运行特定测试
pytest Test/test_feature.py -v
```

## 🤖 AI 助手交互要点

### 标准流程

1. **确认理解** - 复述需求
2. **提供方案** - 2-3 个选项
3. **遵循流程** - TDD 红-绿-重构
4. **主动测试** - 完成后验证
5. **总结反馈** - 列出更改

### 长任务处理

- 主动创建任务清单
- 逐步执行并汇报
- 保存中断状态
- 提供完整总结

### 文档更新

每次完成任务后询问：
```
📝 是否需要更新技术文档？

建议更新：
- 功能文档
- CHANGELOG
- API 文档
```

## ⚠️ 禁止行为

❌ 跳过测试直接实现功能  
❌ 提交无法通过测试的代码  
❌ 硬编码敏感信息  
❌ 在 main 分支直接提交  
❌ 忽略测试失败警告  
❌ 使用裸 except  
❌ 生产环境开启 Debug  

---

## ✅ 必须执行

✅ 新功能必须包含测试  
✅ 修改后运行相关测试  
✅ 测试失败先修复再提交  
✅ 重要决策记录在文档  
✅ 敏感信息放 .env  
✅ 代码符合 PEP 8  
✅ 函数必须有 Docstring

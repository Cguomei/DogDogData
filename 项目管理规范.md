## 🔴 核心原则（必须严格遵守）

### 1️⃣ 不依赖记忆，依赖文档和 Git
- ✅ **技术文档** - 每次重要操作必须记录到 `docs/` 目录
- ✅ **Git 管理** - 所有代码变更必须提交，版本信息以 Git 为准
- ❌ **禁止** - 不要凭记忆回答问题，必须查询 Git 或文档
- 📌 **原因** - 没有人能清楚记得几天前的每件小事，必须有据可查

### 2️⃣ 专业化管理
- ✅ **项目结构清晰** - 代码、文档、测试、脚本分类存放
- ✅ **便于扩展** - 模块化设计，新功能易于添加
- ✅ **便于维护** - 文档完整，代码规范，问题可追溯
- ✅ **标准化流程** - 统一的开发、测试、发布流程

### 3️⃣ 敏感信息管理
- ❌ **禁止上传** - `.env`、密码、密钥、数据库配置等绝不上传 GitHub
- ✅ **.gitignore** - 确保敏感文件在忽略列表中
- ⚠️ **公开仓库** - GitHub 是公开的，任何人都能看到
- ✅ **检查清单** - 每次提交前检查是否包含敏感信息

### 4️⃣ 新功能开发流程
- ✅ **先讨论方案** - 不要直接开始编码
- ✅ **明确需求** - 确认功能目标、技术方案、影响范围
- ✅ **再决定实施** - 方案确认后开始开发
- ❌ **不要猜测** - 有疑问必须询问，不要自作主张

### 5️⃣ 开发闭环流程
```
开发 → 测试 → 修复 → 再测试 → 完成
```
- ✅ **开发功能** - 实现新功能或修复 bug
- ✅ **立即测试** - 验证功能是否正常
- ✅ **发现问题** - 记录问题，分析原因
- ✅ **修复问题** - 针对性修复
- ✅ **再次测试** - 确认问题已解决，无回归
- ✅ **闭环完成** - 确保问题彻底解决

**禁止行为**：
- ❌ 开发完不测试就认为完成
- ❌ 测试发现问题不修复就提交
- ❌ 修复后不验证就认为解决
- ❌ 跳过任何环节

## 📝 文档管理规范

### 文档创建原则
- ✅ **必须主动创建** - 每次重要操作、功能开发、问题修复后都要记录
- ✅ **格式自主选择** - 根据场景选择 .md 或 .txt
- ✅ **放入专门文件夹** - 必须在 `docs/` 或子目录中
- ❌ **禁止散落** - 不要放在项目根目录或代码目录

### 文档存放规范
```
✅ docs/技术文档.md          # 项目文档
✅ Test/docs/测试说明.txt    # 测试文档
❌ README.txt                # 禁止散落在根目录
❌ Test/快速开始.txt         # 禁止散落在代码目录
```

### 必须更新的文档
1. **版本发布说明** - 每次发布新版本时创建/更新
2. **开发文档** - 版本号、技术栈变更、项目结构变化
3. **产品需求文档** - 功能变更、业务规则调整
4. **业务流程文档** - 流程变更、规则更新
5. **验收标准文档** - 验收条件变化

### 更新时机
- ✅ 代码提交前：确保文档与代码同步
- ✅ 版本发布时：编写版本发布说明
- ✅ 功能完成后：更新相关技术文档
- ✅ 问题修复后：记录到修复日志

## 🔧 Git 管理规范

### 提交信息规范
- ✅ **简洁明了** - 只写版本号，如 `v4.5.5`
- ✅ **详细信息** - 写在版本发布文档中
- ❌ **禁止冗长** - 不要在 commit message 中写详细内容

**示例**：
```bash
# ✅ 正确
git commit -m "v4.5.5"

# ❌ 错误
git commit -m "feat: 添加 XXX 功能，修复 YYY 问题，优化 ZZZ..."
```

### 版本标签规范
- ✅ **创建标签** - 每次版本发布都打 tag
- ✅ **推送标签** - `git push origin --tags`
- ✅ **标签描述** - 简短说明版本内容

**示例**：
```bash
git tag -a v4.5.5 -m "v4.5.5 - XXX 功能"
git push origin v4.5.5
```

### 分支管理
- ✅ **main 分支** - 稳定版本，随时可部署
- ✅ **feature 分支** - 新功能开发
- ✅ **及时合并** - 功能完成后合并到 main

## 🧪 测试管理规范

### 测试执行流程
```
1. 开发功能
2. 编写测试用例
3. 运行测试
4. 修复失败用例
5. 再次测试
6. 提交代码
```

### 测试覆盖率要求
- ✅ **API 测试** - 核心接口 100% 覆盖
- ✅ **UI 测试** - 关键页面覆盖
- ✅ **E2E 测试** - 完整业务流程覆盖
- ✅ **总体目标** - 80%+ 代码覆盖率

### 测试运行命令
```bash

# 每日构建

```
